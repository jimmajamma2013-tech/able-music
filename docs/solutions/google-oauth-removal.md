# Solution: Google OAuth button — wrong auth method
**Commits:** `e0bd543` (landing), `097a720` (start.html reference), sessions 3+5
**Date:** Multiple sessions (reoccurred twice)
**Files affected:** `landing.html`, `start.html`

---

## Problem

Google OAuth "Sign in with Google" buttons appeared in:
1. `landing.html` hero section — a prominent "Sign in with Google" CTA alongside the magic link input
2. `start.html` — an OAuth reference in the wizard completion flow

Both were wrong. ABLE uses **magic link email authentication only**. Google OAuth was never part of the product spec.

The button appeared twice across separate sessions — meaning it was regenerated by an AI agent that saw an auth context and defaulted to the Google OAuth pattern (industry standard).

## Root cause

1. **No constraint in the immediate context**: The agent building the auth section didn't have the magic-link-only rule loaded. CLAUDE.md documents it (§2.7), but that file isn't always read on auth-related tasks.
2. **Industry default pattern**: Google OAuth is so common in web apps that it gets generated by default. It requires explicit constraint to suppress.
3. **Unused CSS left behind**: After the button was removed, the Google OAuth CSS rules stayed in `landing.html` for several sessions — a maintenance risk.

## Solution

1. Removed both "Sign in with Google" buttons entirely (no replacement — magic link is the only auth)
2. Removed orphaned Google OAuth CSS from `landing.html`
3. Added the rule explicitly to CLAUDE.md working rules section
4. Added "Never introduce OAuth" to KNOWLEDGE.md hallucination guard section

## Pattern

**Rule:** ABLE uses magic link email auth only. Permanently. Not a V1 scope limit. Not planned for V2. Never introduce Google OAuth, Apple "Sign in with Apple", or any other OAuth provider.

**Reasons for this product decision:**
- Magic link is simpler — artists have email, not necessarily a Google account they want to share
- No dependency on Google/Apple terms, branding, or API uptime
- Clean data model — artist identity is their email, period
- Aligns with ABLE's anti-platform-dependency ethos

**Rule:** When adding any auth-related UI, check: does this introduce any button other than the magic link email flow? If yes — remove it.

**Rule:** After removing a feature, also remove its CSS. Orphaned CSS for removed features accumulates confusion and file size.
